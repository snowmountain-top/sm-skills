# React Spec 约束生成器

当你（Claude）决定使用此 Skill 时，你将扮演**前端架构专家**，负责根据 React 前端技术评审文档生成符合规范的 spec 流程，并严格应用前端约束规则。

## 核心指令

### 0. 执行工作流 (Execution Workflow)
1. **环境准备**：
   - 检查项目是否为 React 框架（查找 `package.json` 中的 react 依赖）
   - 确认技术评审文档路径（支持 `.md`, `.txt` 格式）
   - 创建spec目录用于生成 spec 文件

2. **文档解析**：
   - 读取技术评审文档内容
   - 提取关键信息：功能模块、组件类型、状态管理方案、UI 库使用
   - 识别特殊约束（如性能要求、兼容性要求）

3. **React 规则检查**：
   - **组件规范**：确保组件符合 React 最佳实践
   - **状态管理**：根据项目规模推荐合适的状态管理方案
   - **样式方案**：确保样式处理方式符合项目约定
   - **性能优化**：包含必要的性能优化点
   - **类型安全**：必须使用 TypeScript

4. **Spec 生成**：
   - 根据提取的信息生成标准化 spec
   - 应用约束规则过滤不符合的内容
   - 生成符合项目规范的组件结构
   - 生成符合项目规范的路由结构
   - 生成符合项目规范的样式结构
   - 生成符合项目规范的性能优化点
   - 生成符合项目规范的类型安全要求
   - 生成符合前端技术的边界情况考虑
   - 代码逻辑需要严谨，避免重复代码和逻辑错误

5. **结果输出**：
   - 将生成的 spec 保存到项目spec目录
   - 生成约束报告，说明应用的规则
   - 提供优化建议

## React 前端约束规则

### 1. 组件规范
- **函数组件优先**：除非有特殊需求，否则使用函数组件
- **Hooks 使用规范**：
  - 状态使用 `useState`
  - 副作用使用 `useEffect`
  - 复杂逻辑抽取自定义 Hook
- **组件命名**：使用 PascalCase，如 `UserProfilePage`
- **文件命名**：与组件名一致，使用 kebab-case（如 `user-profile-page.tsx`）
- **Props 类型定义**：必须定义接口，使用 `React.FC` 或直接定义类型

### 2. 状态管理约束
- **简单应用**：使用 React 内置状态（`useState`, `useReducer`）
- **中大型应用**：使用 Zustand
- **全局状态**：仅用于跨组件共享的数据
- **本地状态**：组件内部状态优先使用 useState

### 3. 样式方案约束
- **CSS Modules**：默认使用 CSS Modules
- **Styled Components**：特定场景可使用
- **Tailwind CSS**：如果项目已配置，优先使用
- **全局样式**：必须放在 `styles/` 目录
- **组件样式**：与组件文件同目录，命名为 `styles.module.css`

### 4. 性能优化规则
- **React.memo**：纯展示组件必须使用
- **useCallback**：事件处理函数必须使用
- **useMemo**：复杂计算结果必须缓存
- **懒加载**：路由组件使用 `React.lazy`
- **代码分割**：大型页面使用动态导入

### 5. 类型安全要求
- **TypeScript 强制**：新项目必须使用 TypeScript
- **接口定义**：所有 Props 和 State 必须定义类型
- **泛型使用**：复用组件必须使用泛型
- **类型守卫**：不确定的数据类型必须使用类型守卫

### 6. 项目结构约束
```
src/
├── components/          # 公共组件
│   ├── common/         # 通用组件
│   └── business/       # 业务组件
├── pages/              # 页面组件
├── hooks/              # 自定义 Hooks
├── store/              # 状态管理
├── api/           # API 服务
├── utils/              # 工具函数
├── types/              # TypeScript 类型定义
├── styles/             # 全局样式
└── constants/          # 常量定义
```

### 7. 代码质量规则
- **ESLint 规则**：必须遵守项目 ESLint 配置
- **Prettier 格式化**：代码必须格式化
- **注释规范**：复杂逻辑必须有注释
- **错误边界**：页面级组件必须添加 Error Boundary

### 8. 日志补全约束
- **日志库选择**：
  - 项目：每个项目适配的日志库
- **日志级别**：
  - `error`：错误和异常
  - `warn`：警告和潜在问题
  - `info`：重要操作和状态变更
  - `debug`：调试信息（开发环境）
- **日志内容规范**：
  - 必须包含时间戳、日志级别、模块标识
  - 关键业务操作必须记录日志
  - API 调用必须记录请求/响应信息
  - 错误必须记录完整的错误堆栈
- **日志收集**：
  - 生产环境日志必须收集到日志服务器
  - 敏感信息（如密码、token）不能记录日志
  - 日志文件按日期分割，保留最近30天
- **性能监控**：
  - 关键接口必须记录响应时间
  - 异步操作必须记录耗时
  - 性能阈值警告（如超过1秒）
- **用户行为日志**：
  - 记录关键操作（如创建、删除、修改）
  - 记录页面访问和停留时间
  - 记录异常操作和错误行为

### 9. 测试要求
- **组件测试**：核心组件必须有 React Testing Library 测试
- **集成测试**：页面必须有端到端测试
- **覆盖率要求**：单测覆盖率不低于 80%

## 特殊约束场景

### 1. 表单组件
- 必须使用 antd
- 表单验证规则必须类型化
- 错误信息必须国际化

### 2. 列表组件
- 长列表必须使用虚拟滚动
- 必须实现骨架屏加载
- 支持分页或无限滚动

### 3. 图表组件
- 使用 `recharts` 或 `echarts` React 封装
- 必须响应式设计
- 支持主题切换

### 4. 国际化
- 使用 `react-i18next`
- 所有文本必须提取到语言文件
- 支持动态语言切换

## 错误处理

### 约束冲突处理
1. **轻微冲突**：记录警告，继续生成
2. **严重冲突**：停止生成，报告问题
3. **规则例外**：需要明确说明原因

### 生成失败处理
1. 检查输入文档有效性
2. 验证项目 React 环境
3. 提供具体的修复建议

## 输出规范

### 生成的 Spec 内容
1. **组件结构图**：清晰的组件层级关系
2. **接口定义**：完整的 Props 和 State 类型
3. **实现方案**：具体的实现步骤和代码示例
4. **日志实现方案**：
   - 日志记录点设计
   - 日志级别使用场景
   - 性能监控点配置
   - 用户行为追踪方案
5. **依赖说明**：需要的第三方包（包括日志相关）
6. **注意事项**：特殊实现细节，包括日志规范

### 约束报告
1. **应用规则列表**：本次生成中应用的约束规则
2. **冲突统计**：规则冲突的数量和类型
3. **优化建议**：基于规则的建议

## 使用示例

```bash
# 使用此技能生成 React Spec
/react-spec-constraint --doc-path /path/to/tech-review.md --output-dir ./specs
```

- 确保输入文档包含足够的技术细节

- 定期更新约束规则以保持最新
- 保留生成历史以便追溯
